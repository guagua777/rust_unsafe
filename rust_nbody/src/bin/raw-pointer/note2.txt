Rust 原始指针：什么时候用 / 绝对不要用
这是 Rust 最关键的安全边界，我给你讲最清晰、最实用、能直接写进代码规范的版本。
一、什么时候 应该 / 必须 使用原始指针？
只有少数底层场景才需要原始指针，业务代码永远不需要。
1. 和 C/C++ 交互（FFI）—— 必须用
这是原始指针最常见、最正当的用途。
C 函数传过来的就是指针，Rust 只能用 *mut T / *const T 接收。
rust
运行
// C 代码：void set_value(int* value);
extern "C" {
    fn set_value(v: *mut i32);
}

let mut x = 10;
unsafe { set_value(&mut x) }; // 必须传原始指针
2. 自己写安全抽象 / 底层数据结构
比如你要手写 Vec、LinkedList、Slab、Pool 这种库。
Rust 安全引用无法表达 “自引用”“手动管理内存”，必须用原始指针。
3. 嵌入式 / 操作系统 / 硬件编程
访问固定内存地址、寄存器、物理地址，只能用原始指针。
rust
运行
let UART_ADDR = 0x1000_0000 as *mut u8;
unsafe { *UART_ADDR = b'A'; }
4. 性能极致优化：绕过安全检查
极少数热点路径，需要消除边界检查、生命周期检查、引用计数开销。
但 99.9% 业务不需要。
5. 自引用结构（Self-referential structs）
安全引用无法实现，必须用原始指针。
二、什么时候 绝对不要用 原始指针？
下面这些情况，用了就是代码坏味道。
1. 普通业务逻辑 —— 绝对不要用
你写 Web、API、工具、CLI、爬虫、游戏逻辑……
一律用 & / &mut / Box / Arc / Rc。
原始指针 = 手动管理安全，业务代码完全没必要。
2. 想绕过借用检查器报错 —— 绝对不要用
很多新手遇到 Rust 借用报错，就想用原始指针绕过去。
❌ 错误思路
rust
运行
struct Bad {
    ptr: *mut Data, // 为了绕过借用检查
}
✅ 正确做法
用 Arc<Mutex>
用 Rc<RefCell>
用所有权重构
用生命周期
原始指针不会修复你的代码结构，只会掩盖问题并制造崩溃。
3. 只是想存个内存地址 —— 不要用
存地址用 usize 就行，不要用原始指针。
4. 多线程共享 —— 绝对不要用
原始指针 不是 Send/Sync，不保证线程安全。
多线程共享必须用：
Arc<T>
Mutex<T>
RwLock<T>
原始指针跨线程使用 = 数据竞争 = 崩溃 / 脏数据。
5. 你不能保证内存安全时 —— 不要用
只要你不能 100% 确定：
指针非空
指向有效内存
没有重复释放
没有并发修改
那就不要用原始指针。
三、超级清晰总结（背会这张表）
表格
场景	应该用原始指针？	应该用什么？
业务逻辑、Web、应用层	❌ 绝对不要	&, &mut, Box, Arc, Rc
调用 C 语言函数	✅ 必须用	*mut T, *const T
写底层数据结构	✅ 可以用	原始指针
嵌入式 / 操作系统	✅ 必须用	原始指针
多线程共享	❌ 绝对不要	Arc<Mutex>
绕过借用检查报错	❌ 绝对不要	重构代码
想提升性能	❌ 几乎不要	先优化逻辑
四、最精炼的一句话规则（最重要）
** 原始指针 = 只用于底层、FFI、系统编程
上层应用永远用安全指针（引用 + 智能指针）**
